## Architectural ground truth (from a code audit, not the roadmap prose)

<details>
<summary>"Show your math" is mostly UI, not engine work. TaxEstimateResult (services/tax-engine/sr…</summary>

- **"Show your math" is mostly UI, not engine work.** `TaxEstimateResult` (services/tax-engine/src/types.ts) already exposes AGI, taxable income, the SE-tax sub-components (Social Security / Medicare / Additional Medicare / deductible portion), the CTC nonrefundable/refundable split, state pre-credit + credit-applied, and the mileage deduction (miles × rate). The gaps are **bracket-by-bracket detail** (which rates were hit) and the **standard-deduction amount used** — both need a small additive surface on the result type.
- **Navigation is a hand-rolled state machine** in apps/mobile/src/App.tsx (a `screen` string union). New screens (What-if, Platform comparison, PDF preview, Paywall) = add a union member + a render branch. No nav library.
- **Dashboard tax breakdown** is a `LinearGradient` card of non-interactive `breakdownRow` views in DashboardScreen.tsx — making each a `Pressable` that opens a detail sheet is the hook for show-your-math.
- **Premium track is greenfield:** no `react-native-purchases`, `expo-print`, `expo-store-review`, or `react-native-view-shot` installed; no safe-harbor / prior-year-tax logic; expenses are a flat 4-bucket model (parking/tolls/supplies/phone) with no Schedule C mapping.
- **W2 engine shipped and is live** (`W2_JOB_SUPPORT_ENABLED = true`) — the W-4 optimizer can build directly on it.

</details>

## Workstream 0 — Testing infrastructure (cross-cutting, do first)

<details>
<summary>The foundation everything else rides on. Per the new testing directive.</summary>

The foundation everything else rides on. Per the new testing directive.

- **Playwright (web E2E).** Promote the ad-hoc scratchpad Playwright into committed repo infra: `apps/mobile/e2e/` with a config that boots `expo start --web`, plus page objects for onboarding / dashboard / add-entry. Wire into CI as a separate job. First flow: the core loop (onboard → log entry → dashboard updates) as a regression baseline.
- **Maestro (native flows).** Add `apps/mobile/.maestro/` YAML flows run against an iOS simulator dev build. This is the forcing function that finally closes the long-standing native-only verification gap (Alert dialogs, the `DateField` picker, share sheets, biometric lock, IAP sheet). Authored now; executed in Codemagic CI on a simulator — they cannot run in a headless web environment.
- **Convention going forward:** every v1.1 feature lands with unit tests + a Playwright web flow + a Maestro native flow (where it touches native).

</details>

## Phase A — Free-tier features (no external dependencies; fully buildable & verifiable now)

<details>
<summary>Sequenced by dependency and value.</summary>

Sequenced by dependency and value.

1. ✅ **Show your math (audit trail).** Engine: added `AppliedBracket` + `bracketsApplied`/`standardDeductionUsed` on the federal & state results (`applyBracketsDetailed` in bracketMath.ts; flat-tax states expose a single synthetic bracket). UI: each breakdown row is now a `Pressable` opening `BreakdownDetailSheet` (a bottom-sheet `Modal`), fed by the pure, reusable `breakdownDetails.ts` builder. Shipped with engine unit tests (bracketMath/federal/state), a mobile builder test, and a `show-your-math` Playwright flow — all green. Foundational trust feature; also builds the engine-detail surface the What-if simulator reuses. Never paywalled.
2. ✅ **In-app tax literacy.** Shipped: pure `glossary.ts` (16 plain-language term definitions, no jargon/form-numbers — enforced by a unit test). Each `BreakdownDetail` now carries a `terms` list, and `BreakdownDetailSheet` renders a "What these mean" row of tappable term pills that expand a definition inline. Same surface as #1. Covered by glossary + builder unit tests and an extended `show-your-math` Playwright step.
3. ✅ **What-if earnings simulator.** Shipped: new `WhatIfScreen` (reached from a "What if I earned more?" button on the dashboard, wired into App.tsx's `screen` union). Refactored `computeTaxEstimate` to share an `estimateFromAggregate` core, then added `computeWhatIfEstimate` (+ `WhatIfScenario`/`whatIfAggregate`) so the projection runs the *exact same* engine + W2-withholding pipeline on hypothetical inputs. Pre-fills from the year's actuals, recomputes live as you type, and compares the projected set-aside against the real dashboard number. Covered by calculations unit tests + a `what-if` Playwright flow.
4. ✅ **Platform earnings comparison.** Shipped: `comparePlatforms()` in calculations.ts (pure year-scoped aggregation over `Entry.platform` → earnings/expenses/hours/count + after-expense hourly rate, ranked by earnings). New `PlatformComparisonScreen` (ranked cards with proportional earnings bars + "best $/hr" tag) reached from a dashboard "Compare your platforms" insight card that appears once 2+ platforms have entries. Shared `platforms.ts` constants (labels/icons) de-duped from the dashboard. Covered by calculations unit tests + a `platform-comparison` Playwright flow.
5. ✅ **Earnings share card.** Shipped: `react-native-view-shot` + `expo-sharing` installed (SDK-56 versions via `expo install`). `ShareCard` is a self-contained branded summary (fixed colors so the exported PNG looks identical in light/dark); `ShareEarningsModal` captures it via `captureRef` and hands it to the native share sheet. Reached from a dashboard share icon (shown once there are earnings). Capture + share are **native-only** (gated off web, which shows a "preview only" note) — verified on TestFlight, not in Playwright. Web preview + summary covered by a `share-card` Playwright flow. **Native build note:** view-shot ships a standalone `RNViewShot.xcodeproj`, the same class as the old datetimepicker glob bug, but the existing `xcode-project use-profiles --project ios/SetAsideTracker.xcodeproj --warn-only` scoping already handles it — no CI change needed.
6. ✅ **App rating prompt.** Shipped: `expo-store-review` installed. Pure trigger policy in `appReviewPolicy.ts` (`shouldRequestReview` — ask once, on the 5th entry or first catch-up met; unit-tested) split from the native wrapper `appReview.ts` (`maybeRequestReview` — gated to native, persists an "asked once" flag). Fired fire-and-forget from the entry-save handler. The OS dialog itself is native-only → TestFlight/manual verification ([[feedback_native_module_verification_gap]]).

</details>

## Phase B — Premium track (external + sequencing dependencies)

> **Full Phase B sequencing, prerequisites, architecture audit, and testing strategy now live in [PHASE_B_EXECUTION_PLAN.md](PHASE_B_EXECUTION_PLAN.md).** The summary below is kept for continuity.

<details>
<summary>Operational prerequisite (user's manual steps): stand up a real Sentry project + DSN and…</summary>

**Operational prerequisite (user's manual steps):** stand up a real Sentry project + DSN and an analytics vendor (PostHog/Amplitude) wired into the existing `errorReporting.ts` / `analytics.ts` scaffolds. The plan requires this to land before premium ships.

1. **RevenueCat / IAP payment infrastructure.** `react-native-purchases`, entitlement check at cold start, offline-trust of the cached entitlement, Paywall screen, restore-purchases. Requires App Store Connect product + RevenueCat dashboard setup (user's steps). Gates every premium feature below.
2. **PDF tax-ready summary export** (`expo-print`) **+ Schedule C category alignment** (remap the 4 expense buckets to real Schedule C lines). Ship together; PDF is the anchor.
3. **IRS-compliant mileage log fields.** Additive `Entry` fields (business purpose, start/end location). Schema-only now; also the data-model prerequisite for v1.3 GPS mileage.
4. **Custom expense categories.** Extend the expense model beyond the 4 fixed buckets.
5. **W-4 withholding optimizer.** Builds on the live W2 engine; suggests additional per-paycheck withholding to cover 1099 liability.
6. **Safe-harbor / Form 2210 calculator.** Greenfield engine module (90%-current / 110%-prior rule); needs prior-year tax input or auto-pull from the app's own 2025 estimate.
7. **Year-over-year insights.** Soft-gated until 2+ tax years of data.

</details>

## Multi-state support — design pass required before build

<details>
<summary>Stays in v1.1 scope, but the build is blocked on a written design pass first. Open questi…</summary>

Stays in v1.1 scope, but the build is blocked on a written design pass first. Open questions to resolve before estimating:

- Gig income is generally sourced to the state where work is physically performed, not residence — does `Entry` need a per-entry "state worked in" field vs. a resident/non-resident split?
- Which state pairs to model first?
- How to handle credits for tax paid to a non-resident state, and reciprocity agreements?
- `TaxProfile` currently has one `state` field and the engine has no income-apportionment concept.

After the design pass, re-decide scope (full build in v1.1 vs. defer).

</details>

## Live-testing v1.1 on TestFlight (without merging to master)

<details>
<summary>Phase A (Show-your-math, tax literacy, What-if) is on the v1.1 branch at app version 1.1…</summary>

Phase A (Show-your-math, tax literacy, What-if) is on the `v1.1` branch at app version **1.1.0** (bumped from master's 1.0.0 so builds stay distinct from the pending v1.0 submission). No new native dependencies were added across Phase A, so the iOS build is unaffected by autolinking/signing.

To get a build on TestFlight: in the Codemagic UI, **Start new build → workflow `SetAsideTracker — iOS TestFlight` → branch `v1.1`**. Manual builds ignore the master-only auto-trigger, so this needs no `codemagic.yaml` change and master stays untouched. The workflow builds the engine `dist/`, typechecks, runs unit tests, prebuilds iOS, signs, and submits to TestFlight (build number auto-incremented).
