## Architectural ground truth (from a code audit, not the roadmap prose)

<details>
<summary>"Show your math" is mostly UI, not engine work. TaxEstimateResult (services/tax-engine/srвЂ¦</summary>

- **"Show your math" is mostly UI, not engine work.** `TaxEstimateResult` (services/tax-engine/src/types.ts) already exposes AGI, taxable income, the SE-tax sub-components (Social Security / Medicare / Additional Medicare / deductible portion), the CTC nonrefundable/refundable split, state pre-credit + credit-applied, and the mileage deduction (miles Г— rate). The gaps are **bracket-by-bracket detail** (which rates were hit) and the **standard-deduction amount used** вЂ” both need a small additive surface on the result type.
- **Navigation is a hand-rolled state machine** in apps/mobile/src/App.tsx (a `screen` string union). New screens (What-if, Platform comparison, PDF preview, Paywall) = add a union member + a render branch. No nav library.
- **Dashboard tax breakdown** is a `LinearGradient` card of non-interactive `breakdownRow` views in DashboardScreen.tsx вЂ” making each a `Pressable` that opens a detail sheet is the hook for show-your-math.
- **Premium track is greenfield:** no `react-native-purchases`, `expo-print`, `expo-store-review`, or `react-native-view-shot` installed; no safe-harbor / prior-year-tax logic; expenses are a flat 4-bucket model (parking/tolls/supplies/phone) with no Schedule C mapping.
- **W2 engine shipped and is live** (`W2_JOB_SUPPORT_ENABLED = true`) вЂ” the W-4 optimizer can build directly on it.

</details>

## Phase A вЂ” Free-tier features (no external dependencies; fully buildable & verifiable now)

<details>
<summary>Sequenced by dependency and value.</summary>

Sequenced by dependency and value.

1. вњ… **Show your math (audit trail).** Engine: added `AppliedBracket` + `bracketsApplied`/`standardDeductionUsed` on the federal & state results (`applyBracketsDetailed` in bracketMath.ts; flat-tax states expose a single synthetic bracket). UI: each breakdown row is now a `Pressable` opening `BreakdownDetailSheet` (a bottom-sheet `Modal`), fed by the pure, reusable `breakdownDetails.ts` builder. 2. вњ… **In-app tax literacy.** Shipped: pure `glossary.ts` (16 plain-language term definitions, no jargon/form-numbers вЂ” enforced by a unit test). Each `BreakdownDetail` now carries a `terms` list, and `BreakdownDetailSheet` renders a "What these mean" row of tappable term pills that expand a definition inline. Same surface as #1. Covered by glossary + builder unit tests and an extended `show-your-math` Playwright step.
3. вњ… **What-if earnings simulator.** Shipped: new `WhatIfScreen` (reached from a "What if I earned more?" button on the dashboard, wired into App.tsx's `screen` union). Refactored `computeTaxEstimate` to share an `estimateFromAggregate` core, then added `computeWhatIfEstimate` (+ `WhatIfScenario`/`whatIfAggregate`) so the projection runs the *exact same* engine + W2-withholding pipeline on hypothetical inputs. Pre-fills from the year's actuals, recomputes live as you type, and compares the projected set-aside against the real dashboard number. Covered by calculations unit tests + a `what-if` Playwright flow.
4. вњ… **Platform earnings comparison.** Shipped: `comparePlatforms()` in calculations.ts (pure year-scoped aggregation over `Entry.platform` в†’ earnings/expenses/hours/count + after-expense hourly rate, ranked by earnings). New `PlatformComparisonScreen` (ranked cards with proportional earnings bars + "best $/hr" tag) reached from a dashboard "Compare your platforms" insight card that appears once 2+ platforms have entries. Shared `platforms.ts` constants (labels/icons) de-duped from the dashboard. Covered by calculations unit tests + a `platform-comparison` Playwright flow.
5. **Earnings share card.** Needs `react-native-view-shot` + `expo-sharing` (already installed). Renders a shareable summary image. Native share path в†’ Maestro flow.
6. **App rating prompt.** Needs `expo-store-review`; trigger after the 5th entry / first catch-up met. Native в†’ Maestro-verified.

## Phase B вЂ” Premium track (external + sequencing dependencies)

<details>
<summary>Operational prerequisite (user's manual steps): stand up a real Sentry project + DSN andвЂ¦</summary>

**Operational prerequisite (user's manual steps):** stand up a real Sentry project + DSN and an analytics vendor (PostHog/Amplitude) wired into the existing `errorReporting.ts` / `analytics.ts` scaffolds. The plan requires this to land before premium ships.

1. **RevenueCat / IAP payment infrastructure.** `react-native-purchases`, entitlement check at cold start, offline-trust of the cached entitlement, Paywall screen, restore-purchases. Requires App Store Connect product + RevenueCat dashboard setup (user's steps). Gates every premium feature below.
2. **PDF tax-ready summary export** (`expo-print`) **+ Schedule C category alignment** (remap the 4 expense buckets to real Schedule C lines). Ship together; PDF is the anchor.
3. **IRS-compliant mileage log fields.** Additive `Entry` fields (business purpose, start/end location). Schema-only now; also the data-model prerequisite for v1.3 GPS mileage.
4. **Custom expense categories.** Extend the expense model beyond the 4 fixed buckets.
5. **W-4 withholding optimizer.** Builds on the live W2 engine; suggests additional per-paycheck withholding to cover 1099 liability.
6. **Safe-harbor / Form 2210 calculator.** Greenfield engine module (90%-current / 110%-prior rule); needs prior-year tax input or auto-pull from the app's own 2025 estimate.
7. **Year-over-year insights.** Soft-gated until 2+ tax years of data.

</details>

## Multi-state support вЂ” design pass required before build

<details>
<summary>Stays in v1.1 scope, but the build is blocked on a written design pass first. Open questiвЂ¦</summary>

Stays in v1.1 scope, but the build is blocked on a written design pass first. Open questions to resolve before estimating:

- Gig income is generally sourced to the state where work is physically performed, not residence вЂ” does `Entry` need a per-entry "state worked in" field vs. a resident/non-resident split?
- Which state pairs to model first?
- How to handle credits for tax paid to a non-resident state, and reciprocity agreements?
- `TaxProfile` currently has one `state` field and the engine has no income-apportionment concept.

After the design pass, re-decide scope (full build in v1.1 vs. defer).

</details>
